# 长期项目接管与交接

## 接管前检查

1. 确认当前目录和仓库身份，检查当前分支、远程、最近提交、ahead/behind、未提交和未跟踪文件。
2. 依次阅读根目录及适用子目录的 `AGENTS.md`、README、项目记忆、交接、计划、规格、架构和变更文档。
3. 运行仓库规定的状态、验证和安全检查。
4. 区分用户现有修改与本次工作；不要覆盖、重置、暂存或提交无关修改。
5. 对照文件和 Git 历史核实文档状态，不盲信过期记录。

若项目已有等价文件（例如 `docs/PROJECT_HANDOFF.md`），继续维护现有事实来源，不为满足固定文件名制造重复文档。只有新项目默认使用 `PROJECT_MEMORY.md`、`PLANS.md` 和 `CHANGELOG.md`。

## 里程碑计划

为每个里程碑记录：

- 目标和工作范围；
- 主要文件和依赖；
- 完成标准；
- 测试或验证方法；
- 风险；
- 状态：未开始、进行中、已完成、受阻或已取消；
- 是否已获得提交和推送授权。

一次只推进一个主要里程碑。计划变化时更新原因，不把原计划伪装成已完成。

## 项目记忆

维护一份简洁的当前状态，而不是无限追加流水账。至少记录：

- 总体目标、当前分支和开发阶段；
- 最近完成、正在进行和未完成内容；
- 已知问题、技术债务和关键决定及原因；
- 最近验证及真实结果；
- 最新有效提交与远程同步状态；
- 下一位 Agent 的第一步和优先阅读文件；
- 不应重复尝试或随意修改的事项。

删除失效的临时状态；需要保存历史时移入变更记录或决策文档。

## 提交与推送授权

把“修改文件”“创建本地提交”“推送远程”“创建公开仓库”视为不同权限层级。

- 用户只要求分析或规划：不修改、不提交、不推送。
- 用户要求实现：可修改并验证；除非明确要求或仓库规则授权，不自动提交或推送。
- 用户明确要求阶段提交：按稳定工作单元创建本地提交。
- 用户明确要求阶段推送或给出长期持续授权：在每个重要稳定里程碑和会话结束时安全推送。
- 发现分叉、陌生远程更新、受保护分支或认证异常：停止推送并报告。

## 会话结束交接

即使总任务未完成，也要在授权范围内完成：

1. 检查最终差异和 Git 状态。
2. 运行本次修改相关验证。
3. 更新当前状态、计划、变更记录和必要的 README。
4. 确认重要文件没有遗漏在未跟踪状态。
5. 提交或推送已获授权且稳定的进度。
6. 记录失败步骤、错误摘要、已尝试方法、本地状态和下一步。

交接报告必须明确区分：已完成、已验证、已提交、已推送、未完成和未验证。
