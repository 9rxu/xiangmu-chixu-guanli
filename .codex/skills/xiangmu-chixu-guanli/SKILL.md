# 项目持续管理

把仓库作为可跨环境恢复的项目事实来源。不要让关键目标、状态、决定或下一步只存在于聊天记录中。

## 选择工作模式

- 创建新项目：读取 [references/repository-creation.md](references/repository-creation.md)，再按其中顺序执行。
- 接管、继续或保存项目：读取 [references/project-continuity.md](references/project-continuity.md)。
- 创建或重写项目介绍：同时读取 [references/repository-creation.md](references/repository-creation.md) 的 README 规则。

若项目存在 `AGENTS.md`，先遵守其中更具体的仓库规则。发现与本 Skill 冲突时，采用更安全且更项目专属的规则，并在报告中说明。

## 核心流程

1. 检查工作目录、Git 状态、远程、当前分支、最近提交和未跟踪文件。
2. 阅读仓库现有的 README、Agent 指令、交接、计划、架构和变更文档。
3. 保留来源不明的用户修改；不要为了清理状态而删除、覆盖或重置文件。
4. 根据实际状态选择创建新项目、接管已有项目或执行检查点。
5. 将长期目标拆成可独立验证的里程碑，并把计划写入仓库，而非只写在聊天里。
6. 每完成一个稳定工作单元，运行相关验证并同步项目状态文档。
7. 仅在用户已明确授权提交或项目规则提供持续授权时创建提交；仅在明确授权且远程安全时推送。
8. 结束前重新检查 Git 状态，输出可由下一位 Agent 直接执行的交接报告。

## 新项目标准文件

调用创建新项目模式时，必须创建一个新的独立项目目录、独立 Git 仓库和新的 GitHub 仓库。若当前目录已经位于某个 Git 仓库中，把新项目创建为该仓库之外的同级目录或用户指定的其他目录；禁止把新项目嵌套在现有仓库中。创建前解析并比较两个目录的绝对路径，确认新目录不在现有 Git 根目录之下。

使用 `assets/` 中的模板创建并根据真实项目填写：

- `README.md`
- `AGENTS.md`
- `PROJECT_MEMORY.md`
- `PLANS.md`
- `CHANGELOG.md`

根据技术栈生成 `.gitignore`。不要提交密钥、令牌、Cookie、浏览器状态、个人数据、日志、缓存、构建产物或本地环境文件。未经用户明确选择，不要创建许可证。

把本 Skill 的完整目录复制到新仓库的 `.codex/skills/xiangmu-chixu-guanli/`，确保所有引用和模板仍可用。不要只复制会产生失效链接的单个 `SKILL.md`。

## 检查点规则

在以下情况执行检查点：

- 完成一个可运行、可测试或可独立解释的功能；
- 完成里程碑或重要修复；
- 即将进行高风险或大范围修改；
- 上下文很长、额度可能耗尽或任务可能中断；
- 切换账号、设备、模型或 Agent；
- 用户要求保存进度或当前会话即将结束。

检查点必须包括：

1. 检查差异和未跟踪文件。
2. 运行与本阶段相称的测试、构建、静态检查或格式检查。
3. 修复本阶段引入的问题；无法修复时记录准确失败信息。
4. 更新项目记忆、计划和实际变更记录；行为或使用方式变化时更新 README。
5. 在授权范围内创建一个对应完整工作单元的提交。
6. 在稳定里程碑或会话结束时，在授权和远程安全条件满足后推送。

不要为每一行修改提交，也不要把互不相关的修改塞进同一提交。

## Git 与外部状态安全

- 不强制推送，不改写共享历史，不擅自删除远程仓库、分支或标签。
- 不覆盖同名仓库，不直接修改受保护主分支。
- 推送前获取远程状态；发现分叉、陌生提交或权限问题时停止并报告。
- GitHub 可见性未指定时，新建仓库默认使用私有，并明确报告。
- 未实际成功时，不声称测试、提交、推送或远程创建成功。
- 认证、网络或权限失败时，保留安全的本地成果并写入交接记录。
